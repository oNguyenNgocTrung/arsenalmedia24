135
We’ve defined the
pulse
animation to make a component look bigger for a moment and then return
to its initial size. The animation is triggered by adding a
basket-animate
class to a
basket-line
element. Let’s have a closer look at our
PoSRow
component.
Whether it renders with or without
basket-animate
is determined by its state.
render: ->
product = @props.product
animationClass = {true: 'basket-animate', false: ''}[@state.animate]
div
className: "row basket-line #{animationClass}"
# ...
Because of its initial state at the beginning it is rendered without that class.
getInitialState: ->
{animate: false}
The animation is triggered by new properties coming from the parent component. The product
attributes important for the entire application are kept at a higher level and passed down as
properties. If not for the animations, the component could be completely stateless. We don’t expose
(to the rest of the world) the component’s internal state that is only there for the sake of animating.
When the new properties passed to the component have different
quantity
compared to the
previous value, we trigger a state change. This indicates that we want to be animating.
componentWillReceiveProps: (newProps)->
if newProps.product.quantity != @props.product.quantity
@setState(animate: true)
When the component is re-rendered (updated) it will have a
basket-animate
class added and the
browser will start the
pulse
animation that was shown.
