So the whole process looks like this:
• The button is pressed, so the quantity is changed, for example from
0
to
1
.
• Based on the key definition (
"#{product.id}-#{product.quantity}"
) the div with key
0-0
is considered to
leave
. This can take effect immediately if the element is not animated at
the moment (remember that
CSSTransitionGroup
will only remove the element from the
real DOM if it is not animated).
• The div with key
0-1
is considered to be
entering
. So the
basket-enter
CSS class will be
added, and a moment later the
basket-enter-active
is added as well. This triggers the CSS
transition animation.
• When the animation is finished, the
basket-enter
and
basket-enter-active
classes are
removed. In our case this triggers another short animation.
As you can see in the video, it works. However this approach is far from perfect. There are a couple
of problems with it:
• As the documentation says, this addon should be used for cases
when children are declaratively
added or removed
. We don’t need to add and remove a basket line just because it changed
its quantity. But we are doing it to trigger an animation just because of the limitations of
CSSTransitionGroup
. This is not most performant solution because instead of reusing same
real div in the browser we remove the old one, and add a new one just because the quantity
changed. When the content of a div is small enough it might not matter, but with bigger div’s
you could hurt performance.
