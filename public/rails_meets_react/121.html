With the example above you can see that the whole state is stored inside the top-level component.
All event handlers are also handled by this component. All components above (not shown in this
example) can be stateless. This root component is of course a facade for all calls for UI changes from
the external world - in this case, the external world is the AJAX call for fetching the last user’s saved
session.
This solution has pros and cons I’d like to discuss.
Pros:
•
You are keeping the modified state close to the methods which modify it.
That means you
can work on the state directly and you keep it in one place. It’s easier to track consequences of
calling methods on root component since the only affected state is the local state of the root
component.
•
You produce many stateless components.
That means you can easily reuse them in any part
of your application. They are also easier to work with since they are static.
•
There are easy rules how to bind events inside components tree.
They are all handled in
the root component. Also it forces us to write properties for handlers inside our lower-level
components which makes our code more configurable by default.
Cons:
•
You make the root component big.
It can be hard to reason about code which is big.
Root components written that way tends to have many methods for all handlers, component
initialization methods, facade methods and so on. It can be hard to solve this problem without
introducing a more sophisticated messaging mechanism to your architecture. We use RxJS for
this - we are passing streams via properties to root component and lower-level components
are listening for inputs and producing outputs using these streams.
•
Render is not as efficient as it can be.
Re-rendering causes to create Virtual DOM for the
whole component’s tree. While changes to DOM are always as minimal as possible due how
React works, React must compare the virtual DOM of the whole component’s tree every time.
In environments where performance is necessary it might be faster to compare smaller virtual
DOM trees for re-rendering.
•
Root component is not coherent because it has state and dynamic code for the whole
component tree.
In a bigger components tree it would be more appropriate to distribute state
and code to components that really need this code and state.
