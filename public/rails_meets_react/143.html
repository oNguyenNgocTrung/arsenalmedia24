But that itself would not be enough to animate element for the second time. At the end of the
animation the
basket-animate
class needs to be removed so that changing the quantity would
result in adding the class again, triggering another animation.
To detect the end of animation you need to find the real DOM node that was rendered by using
findDOMNode
and subscribe to the
animationend
event (this is not yet fully standardized, so
webkitAnimationEnd
might be required instead)
When the animation is finished, we get the event, so we can change the state to remove the
basket-
animate
class.
componentDidMount:
->
domNode =
React.findDOMNode(@)

'animationend'
,
@onAnimationEnd,
false
);
domNode.addEventListener(
'webkitAnimationEnd'
, @onAnimationEnd,
false
);
onAnimationEnd:
->
@setState(
animate:
false

If you want to respond in a similar way to CSS transitions instead of CSS animations, the only

)

This is less declarative compared to the previous examples and requires writing the code for
integrating with a browser. But it works without adding and removing real dom nodes so it is more
performant and can be used in even more complicated components.
The component can recognize whether it should animate itself by either comparing incoming
properties or detecting interesting situations in a handler method.
