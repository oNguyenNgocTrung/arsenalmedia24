Tips:
setState
quirks
Let’s discuss two properties of
setState
a little more deeply.
•
setState
is async. It doesn’t update the
@state
immediately but rather schedules such
update to be performed after the handler code is executed. Also updates performed in a single
event handler are
batched together for performance reasons
.
The important takeoff from this is that you should not read
state
after updating it with
setState
in a handler. I know this is pretty inconvenient.
•
setState
updates are shallow, one level deep only.
If
@state
is
warnings: {
one: "one",
two: "two",
}
and you call
setState(warnings: {three: "three"})
you would end up with
warnings: {
three: "three"
}
not:
warnings: {
one:
"one",
two:
"two",
three: "three"
}
This can make certain type of changes very inconvenient to implement in a nice way while still
using
setState
API.
Let’s say you want to update a field with a new value and validate that it is not empty. And you
want to use
setState
to achieve that.
