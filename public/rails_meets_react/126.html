•
flash-enter
,
flash-enter-active
when the component is added; and
•
flash-leave

The first class is added immediately when the given situation occurs. The second class is added in
next tick of the javascript virtual machine (similar to
EventMachine#next_tick
if you are familiar
with that.). In other words, the second class is not added at the same time, but very very quickly
after the first one was added. When the animation has completed, both CSS classes are removed.
What is the purpose of adding those classes? Imagine CSS classes like this:
.flash-appear
{
opacity:
0
.01
;
transition:
opacity
4
s ease-in
;
}
.flash-appear.flash-appear-active
{
opacity:
1;
}
When
flash-appear
is added, the component
opacity
is set to
0.01
, making it almost invisible.
The
transition
property is set to
opacity 4s ease-in
which means
whenever
opacity
changes
don’t apply this change immediately but rather do it gradually over a period of
4s
.
ease-in
is a
timing function
which specifies those gradual, intermediate steps. It can be linear, it can be faster
at the beginning and slower at the end, or the reverse. It be even more complicated. If you are not
familiar with CSS transitions I greatly recommend reading
MDN article on it
.
A short moment later, the
flash-appear-active
class is added which, without
transition
rule
on
flash-appear
, would mean that the
opacity
of
1
should take effect immediately. However,
because of the
transition
it is gradually applied over 4 seconds. The reason we defined
opacity
on
.flash-appear.flash-appear-active
and not just on
.flash-appear-active
is so that it
has a
higher priority
than rules on
.flash-appear
itself.
How does React knows when to remove both classes after those 4 seconds?
Browsers fire an event
when the transition is fully finished.
Remember that
react recognizes components by their keys
.
CSSTransitionGroup
is using the same
heuristic to determine whether the given component is the same one, a new one, or was deleted,
relative to the previous render. You
must
provide keys for the components for it to work properly.
One interesting thing to know is that even if the element was removed from the virtual DOM,
CSSTransitionGroup
will keep the real DOM on the page until the animation is finished. Then
it will be really removed from the real DOM. Of course, this is necessary for the animation to be
seen. But that’s something you gain by using this addon. From the perspective of the virtual DOM
things are simple, and the element is removed. The complicated machinery to handle the animation
is taken care of, and you don’t need to think about it. Also, your component will behave perfectly
even if someone uses it without those animations.
