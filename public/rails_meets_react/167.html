Let’s try to refactor it a little bit.
titleChanged: (event) ->
newState = $.extend(true, {}, @state)
newState.title = event.target.value
@validateTitle(newState)
@setState(newState)
validateTitle: (state) ->
msg = if /\S/.test(state.title) then null else "Cannot be blank"
state.warnings.title = msg
onSubmit: (event) ->
event.preventDefault()
newState = $.extend(true, {}, @state)
@validateTitle(newState)
@validateDate(newState)
@validateDescription(newState)
@setState(newState)
for own key of newState
return if newState.warnings[key]
As you can see, basically whenever there is a possibility that we will change something, we need
to duplicate the state and pass it around, and apply at the end using
setState. We could keep
newState
as instance variable
@newState‘ but I don’t think that would be helpful.
Overcoming the limitations
There are 3 ways you can deal with those problems if you don’t like this approach:
• Mutate the state directly and use
forceUpdate
to tell react that state changed. This is what
we used in the introduction chapter to avoid discussing the limitations of
setState
and focus
on delivering the features.
• Use
immutable data structures
for which
setState
API fits much better.
• Keep state outside React and tell React to re-render manually as well. This is what I would
like to describe in next protip.
