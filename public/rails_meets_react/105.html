Lifecycle Methods
The lifecycle methods allow you to plug into important events that happen during your component’s
life. Since the React workflow is quite unique with its re-rendering cycles, it would be hard to
integrate commonly used jQuery plugins and other external libraries. Fortunately, it is still easy
thanks to the lifecycle methods. They provide sane points where you can inject some code - usually
to integrate with a library or provide an alternative way to mutate the state.
Let’s talk about the basic usage of the lifecycle methods for integrating libraries.
Do not integrate libraries in
render
!
render
should be a side-effect free function - that means, it should return a virtual DOM based only
on state and properties at the given time. You can’t be sure that real DOM is already generated based
on it. That means you can not integrate many front-end libraries (like most jQuery-based plugins)
this way. React can also reject to re-render a component if the real DOM is different than the virtual
DOM representation.
While it may seem inconvenient at the first glance, it has many benefits. Since your
render
return
value relies only on
@props
and
@state
, it is easier to track. Your integration code is separated from
your rendering code. There is the one-to-one relationship between the rendered DOM and virtual
DOM - integration only changes DOM between render cycles.
Integrating an external library inside the
render
method can be hard, if not impossible. Fortunately,
you can still do it in an easy way with tools that React provides.
componentDidMount
/
componentDidUpdate
componentDidMount
and
componentDidUpdate
are methods which are usually used to make
external library integrations.
•
componentDidMount
is called by React once - after your component is rendered for the first
time in a browser. That means it’s going to be called after a direct
React.render
call, or
when your component is rendered due to a change of the parent component’s state.
