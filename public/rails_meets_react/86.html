ref
as a string
Every component has
@refs
property which contains references to mounted child components with
ref
attribute set. The value of
ref
identifies the component in parent’s
@refs
. For example,
ref:
'anInput'
converts to
@refs.anInput
.
So
@refs
are a shoutcut to some of child components. Often they are used for finding the DOM
node from a child component, like this:
ReactDOM.findDOMNode(@refs.focusableContainer)
As said before, you can only access the components that are already mounted. If you want to access
a
ref
as soon as possible, you should use
componentDidMount
lifecycle method (if you are not
familiar, jump to
Lifecycle methods
).
See the whole example, where exactly this mechanism is used. This is
a wrapper
for a casual HTML
input
tag. The wrapper makes sure that after the input is rendered the browser’s focus will be on
it.
FocusableWrapper = React.createClass
componentDidMount: ->
ReactDOM.findDOMNode(@refs.focusableContainer).children[0].focus()
render: ->
React.DOM.div
ref: "focusableContainer"
@props.children
In most cases using
@refs
can be avoided by using natural data-flow mechanisms like callbacks,
props or events. This is why
Refs to Components section of the React docs
comments string-refs as
mostly legacy at this point
.
ref
as a string may be also useful in testing - see
Testing with references
.
DOM node refs
Since React 0.14 refs to DOM components are exposed as the DOM node itself. Note that refs to
custom (user-defined) components work exactly as before; only the built-in DOM components are
affected by this change.
Our
focusableContainer
is a
div
tag, a
built-in DOM component
, not a
custom
one. We don’t need
to use
ReactDOM.findDOMNode(@refs.focusableContainer)
, altough it won’t hurt. Instead,
@refs.focusableContainer
will suffice.
