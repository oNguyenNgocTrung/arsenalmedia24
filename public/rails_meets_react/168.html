Tips: State outside React
Introduction
Over the time your React component can accumulate multiple responsibilities (like we’ve seen in
the Meetup example):
• Rendering the UI
• Handling user interactions
• Business logic
• Communicating with the backend
• Keeping the state
• Manipulating with browser URL and history
• Initializing the widget based on the URL
For small widgets having all of that in one class can still be manageable and sometimes even
beneficial. You don’t need look far away to see how those things affect each other. However such
coupling usually makes testing harder. So the bigger your codebase gets, the more features and use-
cases it needs to support, the more likely you will want to extract those responsibilities into separate
places.
Eventually you will probably reach an architecture inspired by
Flux
and use one of many existing
tools which help you manage it, such as for example
Alt
or
Redux
But before you are there you can extract everything but rendering outside your react components.
Especially the state and handlers. When you do so, you are no longer limited by the
setState
quirks
discussed in the previous tip.
How
Consider React only your View layer responsible for rendering and keep state outside of it. The
CreateNewMeetup
app is responding to events that occur, changing the
@meetup
object and
triggering re-render by directly calling
ReactDOM.render
(which as we know is still using the
effective diff algorithm to perform HTML update).
I personally like this approach because it makes easier later to switch to a different tool if something
better than React shows up.
Yes, you must call
@render()
manually in handlers but we had use
@setState
(or
@forceUpdate
)
to manually trigger re-rendering in React as well so I don’t mind.
