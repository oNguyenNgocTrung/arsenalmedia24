What React does not do is manage your business domain or communication with external services.
In this book we’d like to teach you how to use React in day to day frontend work. Even if you are a
backend developer and you don’t like touching CoffeeScript at all, you may find building UIs very
fast and easier than before. It’s one of the advantages of React over other technologies we’ve tried
before. I hope after reading this book you’ll be as happy with React as we are. Interested? Let’s dive
in to it!
Why React?
There are reasons why we have been using React for over a year. A part of our coding culture is
avoiding coupling with technology that works in magical way. We made a great effort in Rails code
to make our code as explicit as possible - thereby making it understandable and easy to change.
React has its “magic” inside - but it’s not magic that leaks out to code. Code you create with a help
of React is explicit - it’s quite easy to follow paths that may occur during runtime.
Secondly, the architecture of React follows our thinking about an user interface. From my obser-
vations, UI tends to be made of small parts which are often reused on many screens within an
application. Without React I was creating objects which were such parts. Next, I created objects that
composed these small parts into whole views. With React this problem is gone. It is composable by
default. React encourages us to create complex UI solutions by creating small parts and compose it
in a simple, clean way.
The next feature of React is really important. From my experience the most error-prone part of
developing dynamic user interfaces was state transitions. Developers often bypass this problem by
making full redraw of the ‘widget’ when state changes. While it may be OK in most situations, it
may introduce subtle side-effects (like double-binding of browser events - quite a common mistake).
Also, in case when performance matters, it is too slow.
