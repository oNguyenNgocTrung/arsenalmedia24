React makes this state explicit. It handles state transitions automatically, using the same ‘bad’
technique I described - by re-drawing ‘widgets’ when state changes. What it does better is that
it is really fast. React makes changes on an ‘in-memory’ representation of the DOM tree (called
virtual DOM). It uses a smart algorithm to make the least number of changes required. That releases
us from the burden of coding such transitions. It results in faster development speed and minimized
chance to introduce (often recurring) bugs.
The side effect of the feature described above is the
declarativeness
of React components (it is how
React calls ‘parts’ of your user interface). You only need to think about the outcome for a given state.
That makes development much faster. Imagine how frustrating CSS would be if you were forced to
write all the steps on how to change the text color to red using a web browser engine, instead of
simply writing
color: red
…
React may be young, but it is well tested in business environments. Facebook uses it, Instagram’s UI
is written entirely using React. There is a
growing community
around it. We are using it successfully
in our clients’ projects. This makes this technology reliable and you can have confidence it won’t
die anytime soon.
All these pros, combined with speed of development that React provides are good reasons to learn it.
It’ll help you to get on with the challenge of creating sophisticated user interfaces. It’s unlikely that
this trend will die out soon. I personally believe that choosing React was one of the best decisions
our team has made in terms of selecting the right tools to get the job done.
If you are convinced, I’d be really happy to help you to understand this technology and use it in
your projects.
Who should read this book?
This book is aimed mostly at Rails developers. Since I am a Rails developer myself, I use analogies
from that framework. If you are not familiar with Rails, you may still get a lot of value from this
book as React is quite independent from Rails. Most parts are not Rails-specific and you can easily
skip them during your reading.
You can have no experience in React at all - I cover the basics in the first part of this book. We’ll be
using CoffeeScript which is the default scripting language used in the Rails community. If you don’t
know Coffee, I’ll highly recommend to learn at least basics before reading this book.
JSX
On the Internet you may find many examples which use JSX - a little tool to make component
definitions look like HTML.
My opinion is that JSX does not fit well with CoffeeScript. All the examples in this book are written
without JSX. It’s a great improvement in vanilla JavaScript environments though.
